The research behind this library

Every design choice in this library — SHA-pinning, permission scoping, which workflows count as "core" — traces back to patterns observed by sampling public repositories from three organizations. Full writeups with per-repo, per-file citations live in docs/research/ in the repo; this page is the condensed version.

Apple

Sampled ~45 of Apple's most-starred/active public repos via the GitHub API; 57 workflow files read in full. Full writeup: docs/research/apple.md.

Three tiers

  • The Swift library ecosystem (~20 repos) — heavy, standardized Actions usage built on two reusable-workflow hubs: swiftlang/github-workflows and, notably, apple/swift-nio itself acting as a second central CI repo consumed by ~10 sibling repos via workflow_call.
  • Product/platform repos (container, containerization, pkl, servicetalk, foundationdb) — bespoke pipelines, several on self-hosted macOS runners.
  • ML research repos (ml-*) — zero CI. Code drops, not maintained-with-CI projects.

What Apple treats as core

The soundness gate — license headers, formatting, docs-build, API-stability, language hygiene, all bundled in swiftlang/github-workflows' soundness.yml — is the single most consistently applied CI element across Swift repos. Also core: cross-platform build/test matrices spanning Linux distros, Windows, every Apple platform, Static-musl SDK, WebAssembly, Android, and even FreeBSD; and least-privilege permissions + SHA pinning wherever third-party actions are used.

What's supplementary

Security scanning (CodeQL only where individual teams — pkl, servicetalk — chose it; no Scorecard found anywhere in the sample), benchmarking (swift-nio/swift-log only), stale bots (foundationdb/axlearn only), and release automation (per-team styles). And CI itself: the entire ml-* catalog ships with none.

Google

Sampled 38 of Google's most-starred/active public repos; 66 workflow files read in full, plus org-wide code-search counts. Full writeup: docs/research/google.md.

The security layer is the story

GitHub Actions is the default public CI for actively maintained Google OSS (34/38 sampled repos), but internal CI still coexists visibly — Kokoro paths in ~350 files org-wide, gvisor's real pipeline is BuildKite, skia is a LUCI/Gerrit mirror. What's remarkably uniform is the security tooling: OpenSSF Scorecard (39 files), CodeQL (41), step-security harden-runner (64), zizmor (15), SLSA provenance (6) — plus a distinctive org-level semgrep scan of workflow files themselves in google/.github.

SHA-pinning is scorecard-driven, not org-mandated

Repos that adopted Scorecard SHA-pin everything with a version comment and justify every permissions: line in a comment; repos without Scorecard happily use @v4. This library adopts the Scorecard-adopter style as its default across the board, rather than making it conditional.

Dogfooding, everywhere

osv-scanner scans Google's own repos via its own published reusable workflow; oss-fuzz fuzzes its own infra with ClusterFuzzLite; adk-python triages its own PRs/issues/discussions with Gemini-powered ADK agents on a 6-hour cron. This library's own self-check.yml (running reusable-zizmor.yml against this repo's own workflows) follows the same instinct.

GitLab

Studied the live .gitlab-ci.yml tree of gitlab-org/gitlab (the monolith — ~4,400 lines of CI config) plus 8 satellite projects, via the public GitLab API. Full writeup: docs/research/gitlab.md.

A three-tiered reuse model

GitLab's equivalent of a reusable-workflow library is three-tiered: plain include: of local/remote/project files; Jobs/*.gitlab-ci.yml CI/CD templates shipped inside the gitlab codebase itself (Auto DevOps, SAST, Dependency-Scanning); and the newer CI/CD Components Catalog — semver-versioned, input-parameterized building blocks. Every satellite project studied consumes the same danger-review@2.1.0 component and the same Jobs/SAST/Jobs/Dependency-Scanning/Jobs/Secret-Detection templates — a genuinely enforced org-wide standard, not just convention.

Predictive, tiered testing — the most sophisticated pattern found anywhere

GitLab computes a changed-files → matching-tests mapping (Crystalball dynamic mappings, now supplemented by AI-based test selection via its own Duo product), runs a fail-fast subset first, and scales test scope to how many approvals an MR already has (tier-1/2/3). None of this has a clean GitHub Actions equivalent — flagged in the research as a real capability gap, not something this library attempts to replicate.

Security scanning is the most mature of the three orgs studied

SAST/DAST/Dependency-Scanning/Secret-Detection are one-line include: template: references, toggled by CI variables, and mandated org-wide — stronger than Google's broad-but-voluntary adoption and considerably stronger than Apple's sparse, per-team opt-in.

Cross-org comparison

Full writeup: docs/research/comparison.md.

Seven practices where all three organizations independently converge — the strongest signal available that a practice belongs in this library's core tier:

  1. Least-privilege permissions by default, escalated only per-job.
  2. Reusable-workflow centralization over copy-paste — Apple's swiftlang/github-workflows hub, Google's published osv-scanner-action, GitLab's Components Catalog.
  3. Cancel superseded runs, never the mainline — concurrency groups with an explicit protected-branch exception.
  4. Fork-safety guards on anything secret-bearing.
  5. A human/process checkpoint before shipping — draft releases, release PRs, merge-train gates.
  6. Security scanning as a toggle-able template, not a hand-rolled script.
  7. Dogfooding your own tooling in your own CI.

Where they diverge: SHA-pinning is a GitHub-Actions-specific supply-chain concern with no real GitLab analog (GitLab's include: mechanism rarely reaches outside GitLab-run infrastructure); GitLab's predictive/tiered test selection has no GHA-native equivalent; and each org places its heaviest CI somewhere different (Apple: product maturity gates whether CI exists at all; Google: internal infra like BuildKite/ LUCI/Kokoro handles the most demanding projects; GitLab: there's no internal-only shadow system, because GitLab CI/CD is the thing being dogfooded). See the full comparison for what each divergence implies for this library's defaults.
