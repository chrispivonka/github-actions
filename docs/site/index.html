A reusable GitHub Actions workflow library

Built by researching how Apple and Google run CI/CD across their public repositories, and how GitLab structures its own pipelines — then distilling what each treats as core (near-universal, high-leverage) versus supplementary (valuable, adopted case by case) into ready-to-call workflow_call workflows.

View source → Read the research →
Every workflow below is a reusable workflow: you don't copy its code into your repo, you uses: it by reference with a short caller file. Copy-paste caller snippets live in examples/.

Quick start

Pick a workflow, copy its example caller, adjust the inputs. Every reusable workflow's header comment documents the exact permissions: block the caller must grant — copy it as-is; it's already least-privilege.

# .github/workflows/ci.yml
name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  contents: read

jobs:
  ci:
    uses: chrispivonka/github-actions/.github/workflows/reusable-go-ci.yml@main
    permissions:
      contents: read
    with:
      go-versions: '["1.23", "1.24"]'

Pin @main to a tag or commit SHA once this repo starts cutting releases — the way swift-collections pins swiftlang/github-workflows@0.0.12 rather than tracking a branch.

Design conventions

Every workflow in this library follows the same rules, each traceable to a pattern observed in the research:

ConventionWhy
SHA-pin every third-party uses:, with a # vX.Y.Z comment Correlates almost exactly with which repos run OpenSSF Scorecard in the Apple and Google samples — a compromised or force-pushed tag can't silently change what your CI runs.
Least-privilege permissions:, top-level + per-job escalation The majority house style across both orgs' sampled workflows.
persist-credentials: false on every checkout Observed extensively in newer workflows from both orgs; the job token shouldn't outlive the job if it doesn't need to push.
Typed workflow_call inputs for every reasonable override Keeps the library reusable across a caller's actual version matrix / stack, instead of encoding one team's assumptions.
Reusable-workflow centralization One place maintains the CI logic; consumers are thin callers — the swiftlang/github-workflows + apple/swift-nio pattern, and google/osv-scanner-action, applied to this repo's own structure.

Core workflows core

Language-agnostic, near-universal across actively-maintained repos in the research. Adopt these on nearly every repo.

CodeQL core

File: reusable-codeql.yml · Modeled on: apple/pkl, google/gson, google/brotli, google/libphonenumber

Static analysis via CodeQL, with results uploaded as SARIF to code scanning. Supports a matrix of languages — including actions, which scans your workflow files themselves for injection and permission issues, a practice observed in apple/pkl.

InputDefaultPurpose
languagesrequiredJSON array, e.g. '["go", "actions"]'
build-modenonenone / autobuild / manual
queries""Extra query suite, e.g. security-extended
runs-onubuntu-latestRunner image

Required caller permissions: actions: read, contents: read, security-events: write

Recommended trigger: pull_request + push to default branch + weekly schedule

Dependency review core

File: reusable-dependency-review.yml · Modeled on: google/libphonenumber

Fails a PR when it introduces a dependency with a known vulnerability or a disallowed license, by diffing the PR's dependency graph.

InputDefaultPurpose
fail-on-severityhighlow / moderate / high / critical
allow-licenses""SPDX allowlist (mutually exclusive with deny)
deny-licenses""SPDX denylist
comment-summary-in-pron-failurealways / on-failure / never

Required caller permissions: contents: read, pull-requests: write

Required trigger: pull_request (diffs the PR's dependency graph)

OpenSSF Scorecard core

File: reusable-scorecard.yml · Modeled on: google/guava, google/gson, google/brotli, google/magika — 39 hits in the Google sample

The single most consistent security workflow found in the research. Scores your repo's supply-chain security posture, publishes a badge, and uploads SARIF to code scanning.

InputDefaultPurpose
publish-resultstruePublish to the public Scorecard API/badge (public repos only — set false for private repos)
results-fileresults.sarifSARIF output filename

Required caller permissions: read-all

Recommended trigger: branch_protection_rule + weekly schedule + push to default branch

zizmor core

File: reusable-zizmor.yml · Modeled on: google/zx, google/osv-scanner

Statically audits your GitHub Actions workflow files themselves — unpinned actions, script injection, excessive permissions. This repo runs it on its own workflows via self-check.yml, the same dogfooding pattern Google applies to zx and osv-scanner.

InputDefaultPurpose
path.github/workflowsPath to scan
min-severitymediumunknown / informational / low / medium / high
zizmor-version1.22.0Pinned version, installed via uvx
upload-sariftrueUpload results to code scanning

Required caller permissions: contents: read, actions: read, security-events: write (if uploading SARIF)

Stale housekeeping core

File: reusable-stale.yml · Modeled on: google/flatbuffers, apple/foundationdb

Marks inactive issues/PRs stale, then closes them after a grace period.

InputDefaultPurpose
days-before-stale90Inactivity window before marking stale (-1 disables)
days-before-close14Grace period after staleness (-1 never closes)
exempt-labelsnot-stale,pinned,securityComma-separated exempt labels
stale-issue-message / stale-pr-messagegeneric noticeComment posted on staleness
operations-per-run500API-call rate-limit guard

Required caller permissions: issues: write, pull-requests: write

Recommended trigger: daily schedule + workflow_dispatch as a manual escape hatch

PR labeler core

File: reusable-labeler.yml · Modeled on: google/flatbuffers, google/gvisor

Applies labels to PRs based on which paths they touch, using a .github/labeler.yml config in the caller repo.

InputDefaultPurpose
configuration-path.github/labeler.ymlPath to labeler config in caller repo
sync-labelstrueRemove labels when matching files are reverted out of the PR

Required caller permissions: contents: read, pull-requests: write

Required trigger: pull_request_target (needed to label forked PRs)

Security note: pull_request_target runs with base-repo credentials. Google's org-level policy treats it as requiring an explicit exception — this workflow is safe with it only because it never checks out or executes PR code. Don't add steps that do.

Release please core

File: reusable-release-please.yml · Modeled on: google/dotprompt, the google/adk-* family, google/gts

Maintains a running "release PR" from conventional-commit history and cuts a tagged GitHub release on merge. Outputs let you chain a publish job only when a release was actually created.

InputDefaultPurpose
release-typerequirede.g. node, python, go, rust, simple
config-file""Path to release-please-config.json (overrides release-type)
manifest-file""Path to .release-please-manifest.json
target-branch""Branch to track

Outputs: release-created, tag-name, upload-url

Required caller permissions: contents: write, pull-requests: write

Required trigger: push to default branch

GitHub release core

File: reusable-github-release.yml · Modeled on: apple/container, google/jsonnet

Publishes a GitHub Release with build artifacts attached — draft by default, so a human confirms publication, the pattern both Apple and Google favor over releasing automatically on tag push. Doesn't build your artifacts; hand them off via actions/upload-artifact from a prior job.

InputDefaultPurpose
filesrequiredGlob(s) of artifact paths to attach
tag-name""Defaults to the triggering ref
drafttrueRequire manual publish
prereleasefalseMark as prerelease
generate-release-notestrueAuto-generate notes from merged PRs
artifact-name""If set, downloads this build artifact before publishing

Required caller permissions: contents: write

Per-language CI supplementary

Adopt whichever matches your stack — don't stack multiple unless the repo is genuinely polyglot (see google/magika's per-language pattern in the research for when that's warranted). Every one of these takes os and its version-axis input as JSON arrays, so you get a full matrix build for free — the same shape as Apple's swift_package_test.yml, scoped to one language per workflow.

Node.js CI

File: reusable-node-ci.yml · Stages: install → lint → typecheck → test → build
InputDefault
node-versions'["22"]'
os'["ubuntu-latest"]'
package-managernpm (or yarn / pnpm)
working-directory.
run-lint / run-typecheck / run-test / run-buildtrue each — set false to skip a stage your repo doesn't have

Required caller permissions: contents: read

Assumes lint / typecheck / test / build npm scripts in package.json.

Python CI

File: reusable-python-ci.yml · Stages: ruff check → ruff format → pytest, via uv
InputDefault
python-versions'["3.12"]'
os'["ubuntu-latest"]'
working-directory.
run-lint / run-testtrue each
test-args"" — extra pytest arguments

Required caller permissions: contents: read

Assumes a pyproject.toml consumable by uv/ruff/pytest.

Go CI

os'["ubuntu-latest"]'
working-directory.

Rust CI

File: reusable-rust-ci.yml · Stages: fmt --check → clippy -D warnings → test
InputDefault
toolchains'["stable"]'
os'["ubuntu-latest"]'
working-directory.
run-fmt / run-clippy / run-testtrue each

Required caller permissions: contents: read

Uses dtolnay/rust-toolchain + Swatinem/rust-cache — the de facto standard pairing across the Rust OSS ecosystem.

.NET CI

File: reusable-dotnet-ci.yml · Stages: restore → format check → build → test
InputDefault
dotnet-versions'["9.0.x"]'
os'["ubuntu-latest"]'
working-directory.
run-format / run-testtrue each
configurationRelease

Required caller permissions: contents: read

No .NET-heavy repos turned up in the Apple/Google research samples (both orgs skew Swift/Java/C++/Go/Python) — this workflow adapts the same staging pattern used by the other language workflows here rather than citing a specific org example.
